# Chatbot Project

Vietnamese chatbot project with:

- dataset download and preprocessing for UIT-ViQuAD2.0,
- evaluation utilities for QA predictions,
- an integrated Gradio UI for dataset browsing, metrics, evaluation, and Rasa chat,
- a Rasa backend for the assistant itself.

## Knowledge Base

The Rasa knowledge pipeline is built from the processed UIT-ViQuAD2.0 corpus.

- source corpus: `data/processed/corpus.json`
- chunked docs for Rasa indexing: `rasa_app/docs/`
- generated runtime state: `rasa_app/.rasa/`
- trained model archives: `rasa_app/models/`

The vector index itself is generated by Rasa during `rasa train` from the chunked docs in `rasa_app/docs/`. It is not stored as a separate hand-managed file in the repo.

## Local Run

Use this when you want to run the components directly on your machine.

### 1. Install Python dependencies

From the repo root:

```bash
uv sync
```

If you do not use `uv`, install the project dependencies with your own Python environment and package manager.

### 2. Download and preprocess the dataset

This creates `data/raw/` and `data/processed/`, which are required by the UI evaluation tabs.

```bash
python -m data.download
```

### 3. Start the Rasa backend

The Rasa server builds chunked knowledge docs from `data/processed/corpus.json`, trains the model, and exposes the REST webhook used by the UI chat tab.

Create a separate environment for Rasa if you are not using Docker:

```bash
cd rasa_app
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
```

Then start the server:

```bash
python main.py rasa
```

The server listens on `http://localhost:5005`. Keep this running in one terminal while you use the UI or the evaluation command.

### 4. Start the Gradio UI

From the repo root:

```bash
python main.py demo
```

Or run the UI module directly:

```bash
python ui/app.py
```

The UI listens on `http://localhost:7860` by default.

### 5. Optional environment variables

Set these if you need to point the UI at a different Rasa endpoint or port:

```bash
export RASA_REST_URL=http://localhost:5005/webhooks/rest/webhook
export RASA_CHAT_SENDER=gradio_user
export RASA_API_TIMEOUT=30
export UI_PORT=7860
```

### 6. Run the evaluation command

With the Rasa backend still running, you can score the validation split against the knowledge base:

```bash
python main.py evaluate
```

## Docker Run

Docker is the easiest way to run the whole stack together.

### 1. Download the dataset on the host

The UI and Rasa containers mount the host `./data` directory, so the processed dataset must exist locally first.

```bash
python -m data.download
```

### 2. Build and start the stack

From the repo root:

```bash
docker compose up --build
```

This starts:

- `rasa` on port `5005`
- `ui` on port `7860`

The Rasa container also builds the chunked knowledge docs and trains the assistant before it starts serving.

### 3. Open the UI

Visit:

```text
http://localhost:7860
```

## What the UI tabs do

- Dataset tab: search processed UIT-ViQuAD2.0 examples across all splits, including title, question, answers, and passage context
- Metrics tab: compute EM / F1 / BLEU for a single prediction
- Tạo Predictions tab: upload a processed split JSON shaped like `data/processed/test.json` and generate a downloadable `predictions.json` file from live Rasa responses
- Evaluation tab: load a `predictions.json` file and score it against `validation` or `test`
- Chat tab: send messages to the Rasa REST webhook

## Notes

- The evaluation tabs need the processed data files in `data/processed/`.
- In Docker, `docker-compose.yml` mounts `./data:/app/data:ro` for the UI and `./data:/data:ro` for the Rasa container.
- The Rasa backend uses `rasa_app/start-rasa.sh`, which rebuilds the knowledge docs, retrains the assistant, and then runs `rasa run --enable-api --port 5005 --cors "*" --quiet` by default. Set `RASA_DEBUG=1` if you want verbose logs (`-vv`).

## Security Before Push

- Do not commit `rasa_app/.env`. It contains runtime secrets such as Rasa license values and provider API keys.
- Do not commit `rasa_app/.rasa/`. It is generated runtime state and cache data.
- Do not commit generated model archives in `rasa_app/models/`. The repo ignores `rasa_app/models/*.tar.gz`, but already tracked files must still be removed from Git with `git rm --cached`.
- If a real API key has been used locally, rotate it before publishing the repository.
